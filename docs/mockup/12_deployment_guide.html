4 Firebase Setup

Firebase 프로젝트를 생성하고 Firestore, Storage, Auth를 설정하는 상세 과정입니다.

1 Firebase 프로젝트 생성

Firebase Console에 접속하여 새 프로젝트를 생성합니다.

Firebase Console 접속 → 프로젝트 추가 → 프로젝트명: petholo-prod → Google Analytics 활성화 (선택) → 프로젝트 생성 완료
💡
프로젝트 ID는 나중에 변경할 수 없으므로 신중하게 정하세요. 예: petholo-prod
2 Firestore 데이터베이스 활성화

Firebase Console → Firestore Database → 데이터베이스 만들기

Firestore Database 클릭 → 데이터베이스 만들기 → 리전: asia-northeast3 (서울) → 프로덕션 모드 시작
⚠
리전 선택 중요: asia-northeast3 (서울)을 선택하세요. 한번 설정하면 변경할 수 없습니다. 한국 사용자 대상 서비스이므로 지연시간을 최소화합니다.
3 Firebase Storage 활성화

Firebase Console → Storage → 시작하기 → 프로덕션 모드

버킷 위치: asia-northeast3 (Firestore와 동일 리전 권장)

5 Firestore Security Rules 설정

Firebase Console → Firestore → 규칙 탭에서 아래 규칙을 설정합니다.

firestore.rulesFirestore Security Rules
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // 기본: 모든 접근 차단 (서버 SDK는 규칙 무시)
    match /{document=**} {
      allow read, write: if false;
    }

    // users 컬렉션: 본인 문서만 읽기 허용
    match /users/{userId} {
      allow read: if request.auth != null
                  && request.auth.uid == userId;
      allow write: if false;
    }

    // pets 컬렉션: 소유자만 읽기 허용
    match /pets/{petId} {
      allow read: if request.auth != null
                  && resource.data.userId == request.auth.uid;
      allow write: if false;
    }

    // profiles 컬렉션: 소유자 읽기 + 공유 링크 읽기
    match /profiles/{profileId} {
      allow read: if request.auth != null
                  && resource.data.userId == request.auth.uid;
      allow write: if false;
    }
  }
}
💡
PetHolo는 Firebase Admin SDK를 사용하므로 서버 측 접근은 Security Rules의 영향을 받지 않습니다. 위 규칙은 클라이언트에서 직접 Firestore에 접근하는 것을 방지하는 보안 장치입니다.

7 Backend Deployment - Option B: Google Cloud Run

Cloud Run은 컨테이너 기반 서버리스 플랫폼입니다. 트래픽에 따라 자동 스케일링되며, 사용한 만큼만 비용을 지불합니다.

1 Dockerfile 작성 (Multi-stage Build)
Dockerfileapps/api/Dockerfile

2 GitHub Actions Workflow
yaml.github/workflows/deploy.yml
name: PetHolo CI/CD

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

env:
  NODE_VERSION: '20'
  PNPM_VERSION: '9'

jobs:
  # ── Job 1: Lint & Test ──
  quality:
    name: Lint & Test
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v4
        with:
          version: ${{ env.PNPM_VERSION }}

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Lint (Frontend)
        run: pnpm --filter @petholo/web lint

      - name: Build check (Backend)
        run: pnpm --filter @petholo/api build

      - name: Build check (Frontend)
        run: pnpm --filter @petholo/web build
        env:
          NEXT_PUBLIC_API_URL: https://api.petholo.com/api
          NEXT_PUBLIC_GOOGLE_CLIENT_ID: placeholder

  # ── Job 2: Deploy Frontend (Vercel) ──
  deploy-frontend:
    name: Deploy Frontend
    needs: quality
    runs-on: ubuntu-latest
    if: github.event_name == 'push'
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Deploy to Vercel (Production)
        if: github.ref == 'refs/heads/main'
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          vercel-args: '--prod'
          working-directory: apps/web

      - name: Deploy to Vercel (Staging)
        if: github.ref == 'refs/heads/develop'
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          working-directory: apps/web

  # ── Job 3: Deploy Backend (Railway) ──
  deploy-backend:
    name: Deploy Backend
    needs: quality
    runs-on: ubuntu-latest
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Install Railway CLI
        run: npm install -g @railway/cli

      - name: Deploy to Railway
        run: railway up --service petholo-api
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}

1 Winston Logger 설정 (현재 구성)

백엔드는 Winston 로거를 사용하여 구조화된 로그를 출력합니다. 현재 설정:

typescriptapps/api/src/utils/logger.ts (현재 코드)
import winston from 'winston';
import { env } from '../config/env';

const logFormat = winston.format.combine(
  winston.format.timestamp({ format: 'YYYY-MM-DD HH:mm:ss' }),
  winston.format.errors({ stack: true }),
  winston.format.printf(({ timestamp, level, message, stack }) => {
    return `${timestamp} [${level.toUpperCase()}]: ${stack || message}`;
  })
);

export const logger = winston.createLogger({
  level: env.isDev ? 'debug' : 'info',
  format: logFormat,
  transports: [
    // 콘솔 출력 (Railway/Cloud Run 로그에 표시)
    new winston.transports.Console({
      format: winston.format.combine(
        winston.format.colorize(),
        logFormat
      ),
    }),
    // 에러 로그 파일 (5MB x 5개 로테이션)
    new winston.transports.File({
      filename: 'logs/error.log',
      level: 'error',
      maxsize: 5 * 1024 * 1024,
      maxFiles: 5,
    }),
    // 전체 로그 파일 (10MB x 5개 로테이션)
    new winston.transports.File({
      filename: 'logs/combined.log',
      maxsize: 10 * 1024 * 1024,
      maxFiles: 5,
    }),
  ],
});

12 Troubleshooting

배포 후 자주 발생하는 문제와 해결 방법입니다.

❌ CORS 에러 (Access-Control-Allow-Origin)
원인: 백엔드의 CORS_ORIGIN 환경변수에 프론트엔드 도메인이 누락됨
해결:
1. 백엔드 환경변수 확인: CORS_ORIGIN=https://petholo.com,https://www.petholo.com
2. 프로토콜(https://) 포함 여부 확인
3. 끝에 슬래시(/) 가 없는지 확인
4. 환경변수 변경 후 서비스 재시작
❌ Firebase Permission Denied
원인: 서비스 계정 키가 잘못되었거나 권한이 부족함
해결:
1. FIREBASE_PROJECT_ID, FIREBASE_CLIENT_EMAIL, FIREBASE_PRIVATE_KEY 값 정확성 확인
2. FIREBASE_PRIVATE_KEY의 \n 이스케이프 확인 (JSON에서 복사 시 주의)
3. 서비스 계정에 "Firebase Admin SDK Administrator Service Agent" 역할 확인
4. Firebase Console → 프로젝트 설정 → 서비스 계정에서 새 키 재발급
❌ JWT Token Expired / Invalid
원인: 토큰 만료 후 리프레시 플로우 실패 또는 시크릿 불일치
해결:
1. 프론트엔드에서 401 응답 시 자동으로 /api/auth/refresh 호출하는지 확인
2. JWT_SECRET과 JWT_REFRESH_SECRET이 서로 다른 값인지 확인
3. 배포 환경 간 시크릿 값이 동일한지 확인 (스테이징 vs 프로덕션)
4. 토큰 만료시간: 액세스 15분, 리프레시 7일
❌ AI 영상 생성 Timeout / 실패
원인: API 키 만료, 요청 한도 초과, 또는 네트워크 문제
해결:
1. NanoBanana / Kling AI API 키 유효성 확인 (각 서비스 대시보드에서 확인)
2. API 사용량 한도 확인 (Rate Limit / Quota)
3. 백엔드 로그에서 구체적인 에러 메시지 확인: railway logs
4. Cloud Run의 경우 타임아웃 설정 확인 (기본 300초)
❌ Cafe24 Webhook 수신 실패
원인: HMAC 시크릿 불일치 또는 Webhook URL 미등록
해결:
1. Cafe24 관리자에서 등록한 Webhook URL 확인: https://api.petholo.com/api/webhooks/cafe24
2. CAFE24_WEBHOOK_SECRET이 Cafe24 설정과 동일한지 확인
3. Webhook rate limit 확인: 분당 50회 제한
4. 백엔드 로그에서 Webhook 요청 수신 여부 확인
❌ Vercel 빌드 실패
원인: Root Directory 설정 오류, 환경변수 누락, 또는 의존성 문제
해결:
1. Root Directory가 apps/web으로 설정되어 있는지 확인
2. NEXT_PUBLIC_* 환경변수가 Vercel에 등록되어 있는지 확인
3. 로컬에서 pnpm build 성공 여부 먼저 확인
4. Vercel 빌드 로그에서 구체적인 에러 확인
❌ Railway 배포 후 서비스 접속 불가
원인: 포트 설정 오류 또는 Health Check 실패
해결:
1. PORT 환경변수가 4000으로 설정되어 있는지 확인
2. railway.toml의 internalPort가 PORT와 일치하는지 확인
3. Health check path가 /api/health인지 확인
4. Railway 대시보드 → Deployments에서 빌드 로그 확인
❌ 파일 업로드 실패 (Storage)
원인: S3/Firebase Storage 설정 오류 또는 CORS 미설정
해결:
1. STORAGE_PROVIDER가 올바르게 설정되어 있는지 확인 (local / s3)
2. S3 사용 시: S3_ACCESS_KEY, S3_SECRET_KEY, S3_BUCKET 확인
3. Firebase Storage 사용 시: 버킷명과 CORS 설정 확인
4. 업로드 파일 크기 제한: 10MB (Express body parser 설정)
